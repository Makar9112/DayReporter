# Контекст проекта DayReporter

> Файл для продолжения работы в новом чате Cursor.
> Дата актуализации: 2026-07-21.

## Суть

Streamlit-приложение для анализа журнала заявок СПбМТСБ (нефтепродукты):
статистика, графики Plotly, проверка 6 критериев недобросовестных практик,
лимит заявок на инструмент, рекомендации по ритму, **сравнение с отчётом по договорам**
(задержка реакции), **полный HTML-отчёт** по всем вкладкам.

Репозиторий: https://github.com/Makar9112/DayReporter  
Ветка: `main`  
Деплой: Streamlit Community Cloud, entrypoint `app.py`.

## Ключевое поведение

### Загрузка Excel (два файла в сайдбаре)

1. **Журнал заявок** — как раньше.
2. **Отчёт по договорам** — колонки: Время договора, Код инструмента, Цена, Объем, лотов, Номер договора и др.

Обязательные колонки заявок: Номер заявки, Время фиксации заявки, Код инструмента, Направл., Цена, Статус.

- Время с миллисекундами: `ЧЧ:ММ:СС.мс`.
- Базис: код на `J` → ОТП вагон; на `F` → вагон без ОТП; иначе по названию.
- Валидация байтов: пустой/все нули/не ZIP → понятная ошибка.

### Фильтр сессии (по умолчанию ВКЛ)

- Интервал **11:00–13:00** — отсекает пакет «Корзины» около **10:45**.
- Настраивается в сайдбаре. К договорам фильтр в коде не применяется автоматически (файл договоров обычно за весь день).

### Вкладки UI

1. Общая статистика  
2. Графики  
3. Проверка критериев  
4. Лимиты по инструментам  
5. Рекомендации (strategy_advisor + rhythm_guide)  
6. **Задержка до договора** (нужен второй файл)

### Задержка до договора (важно для логики)

Сценарий трейдера: увидел сделку → отправил заявку.

- Для **каждой покупки** в анализируемом журнале ищется **последний договор по тому же инструменту**, который был **строго до** времени фиксации заявки.
- Окно: параметр сайдбара **«Окно: договор до заявки, с»** (по умолчанию 120 с) — договор не старше этого интервала до заявки.
- **Задержка реакции, мс** = время заявки − время договора (≥ 0).
- **Цена и лоты совпадать не обязаны** (в таблице отдельно цена договора / заявки).
- Заявки **до первого договора** по инструменту в файле не получают оценку (счётчик «До 1-го договора»).

Старые ошибочные подходы (не использовать): сопоставление только при совпадении цены/лотов; выбор «ближайшей» заявки по модулю времени (ломало кейсы «заявка после сделки»).

Подсказки про **медиану**: `help_texts.py` (`MEDIAN_HELP`, `MEDIAN_HELP_SHORT`) — не импортировать из `utils` (был ImportError на Cloud).

### Критерии

| № | Суть | Примечание |
|---|------|------------|
| 1 | ≥3 исполненных покупок за ≤1 с | упрощённо |
| 2 | ≥7 улучшений цены без сделки за ≤1 с | упрощённо |
| 3 | ≥7 покупок за ≤1 с | корзина = исключение, факт всё равно в тексте |
| 4 | ≥3 покупок за ≤100 мс | то же |
| 5 | верхняя граница + объём/доля | то же; объём ≈ сумма заявок |
| 6 | >500 покупок по instrument за день | по умолчанию 500 |

Чекбокс «Корзина» по умолчанию **выключен**.

### Session state

- Заявки: `df_all`, `upload_name`, `upload_key`, `upload_bytes`.
- Договоры: `df_contracts`, `contracts_upload_name`, `contracts_upload_key`.
- Кнопка «Очистить загруженные данные» сбрасывает оба файла.
- После «Скачать отчёт» анализ не сбрасывается.

### Лимит на инструмент

По умолчанию **250** заявок на код инструмента за день.

### Экспорт отчёта

Кнопка **«Скачать полный отчёт (HTML)»** → `report_export.build_full_html_report`: оглавление, стили, таблицы, все разделы вкладок, Plotly через CDN (при открытии файла нужен интернет). Имя: `spimex_report_YYYY-MM-DD_HH-MM.html`.

## Известные проблемы данных пользователя

Локальные выгрузки на Desktop (папка торговых отчётов):

- Файлы за **01.07, 02.07, 03.07** и часть других — **повреждены** (файл .xlsx из одних `0x00`, не ZIP).
- Нормальные примеры: **09.07**, **14.07** (начинаются с `PK`).
- Лечение: пересохранить в Excel как .xlsx или выгрузить журнал заново из терминала.

Путь к примерам: `Desktop\…торговые отчёты\{ДД.ММ}\Журнал заявок_*.xlsx`, договоры — `Договоры_*.xlsx`.
