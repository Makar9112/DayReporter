# Контекст проекта DayReporter

> Файл для продолжения работы в новом чате Cursor.
> Дата актуализации: 2026-07-21 (залив, пролив, читаемые графики лимитов).

## Суть

Streamlit-приложение для анализа журнала заявок СПбМТСБ (нефтепродукты):
статистика, графики Plotly, проверка 6 критериев недобросовестных практик,
лимит заявок на инструмент, **залив и проливы по договорам** на вкладке лимитов,
рекомендации по ритму, сравнение с отчётом по договорам (задержка реакции),
полный HTML-отчёт по всем вкладкам.

Репозиторий: https://github.com/Makar9112/DayReporter  
Ветка: `main`  
Деплой: Streamlit Community Cloud, entrypoint `app.py`.

## Ключевое поведение

### Загрузка Excel (два файла в сайдбаре)

1. **Журнал заявок** — обязательные колонки: Номер заявки, Время фиксации заявки, Код инструмента, Направл., Цена, Статус; желательно **Наименование инструмента**.
2. **Отчёт по договорам** — Время договора, Код инструмента, Цена, Объем, лотов, Номер договора; часто **Наименование и Базис поставки** (алиас на «Наименование инструмента» в `contracts_lag`).

- Время с миллисекундами: `ЧЧ:ММ:СС.мс`.
- Базис поставки (колонка «Базис» в заявках): код на `J` → ОТП вагон; на `F` → вагон без ОТП; иначе по названию.
- Валидация `.xlsx`: пустой / нули / не ZIP → понятная ошибка (`utils._validate_excel_bytes`).

### Фильтр сессии (по умолчанию ВКЛ)

- Интервал **11:00–13:00** — исключает «Корзину» ~10:45.
- **Лимиты / залив / HTML-график лимитов:** договоры режутся **`df_contracts_session`** тем же интервалом, что заявки (`filter_by_session_time`).
- **Задержка до договора:** полный файл договоров за день (фильтр сессии не применяется).

### Вкладки UI

1. Общая статистика  
2. Графики  
3. Проверка критериев  
4. **Лимиты по инструментам** (заявки + залив/проливы при загруженных договорах)  
5. Рекомендации (`strategy_advisor` + `rhythm_guide`)  
6. Задержка до договора (нужен второй файл)

### Залив и пролив (вкладка «Лимиты»)

Термины от руководства / практики трейдинга:

| Термин | Смысл в приложении |
|--------|---------------------|
| **Залив** | Суммарный объём сделок базиса по коду инструмента за интервал: лоты (вагоны) и **тонны** (бензин **60 т**/вагон, дизель **65 т** — `wagon_tons_per_lot`). |
| **Пролив** | Число «пачек» по времени: договоры одного кода сортируются по `Время_сec`; если соседние сделки ближе **N мс** (UI по умолчанию **5**, диапазон 1–20) — один пролив. Пример: два договора в `11:12:30.083` → **1 пролив**, **2 договора**. |

Реализация (`contracts_lag.py`):

- `count_proliv_clusters`, `assign_proliv_ids` — кластеризация по времени.
- `aggregate_basis_fill_by_instrument(..., proliv_spread_ms=)` → колонки: Договоров, **Проливов**, Лоты, Тонны залива, Наименование_договора.
- `contracts_detail_for_instrument` — таблица с колонкой **Пролив** (номер пачки).

Графики (`basis_fill_charts.py`):

- `merge_orders_and_basis_fill`, `prepare_limits_chart_frame`, `fig_instruments_limit_with_basis_fill`.
- **По умолчанию:** только коды из **вашего журнала** (`scope=my`), **топ 20**, вид **две панели** (заявки / тонны залива).
- Опции: все инструменты из договоров, топ 10–30, сортировка (активность, заявки, тонны, **проливы**), виды (горизонтальный, две оси Y и др.).
- Без файла договоров — только классический график заявок (`trade_analytics.fig_instruments_limit`).

**Наименования:** приоритет **журнал** — `utils.pick_first_nonempty` по «Наименование инструмента»; из договора — `Наименование_договора` только если в журнале пусто. Подсказки на графике и «код · имя» на горизонтальном виде.

### Задержка до договора

Сценарий: увидел сделку → отправил заявку.

- Для каждой **покупки** — последний договор по тому же инструменту **строго до** фиксации заявки.
- Окно сайдбара «Окно: договор до заявки, с» (по умолчанию 120 с).
- Задержка, мс = время заявки − время договора (≥ 0). Цена/лоты могут не совпадать.
- Не использовать: матч только по цене/лотам; «ближайшая» заявка по модулю времени.

Медиана: `help_texts.py` — не импортировать из `utils`.

### Критерии

| № | Суть | Примечание |
|---|------|------------|
| 1 | ≥3 исполненных покупок за ≤1 с | упрощённо |
| 2 | ≥7 улучшений цены без сделки за ≤1 с | упрощённо |
| 3 | ≥7 покупок за ≤1 с | корзина = исключение |
| 4 | ≥3 покупок за ≤100 мс | то же |
| 5 | верхняя граница + объём/доля | то же |
| 6 | >500 покупок по instrument за день | по умолчанию 500 |

Чекбокс «Корзина» по умолчанию **выключен**. Лимит стакана по умолчанию **250** заявок на код.

### Session state

- Заявки: `df_all`, `upload_name`, `upload_key`, `upload_bytes`.
- Договоры: `df_contracts`, `contracts_upload_name`, `contracts_upload_key`.
- «Очистить загруженные данные» сбрасывает оба файла.
- «Скачать отчёт» не сбрасывает анализ.

### Экспорт отчёта

`report_export.build_full_html_report` — все разделы; график лимитов с заливом: `scope=my`, топ 20, `split_panels`, если передан `df_contracts_session`. Имя файла: `spimex_report_YYYY-MM-DD_HH-MM.html`.

## Cloud / импорты (важно)

## Предпочтения пользователя

- Ответы по-русски.
- Не коммитить `.xlsx`, `.env`, secrets.
- **`CONTEXT.md` обновлять** при заметных изменениях.

### Git: коммит и push (обязательно для ассистента)

После осмысленных изменений в коде/конфиге **без отдельной просьбы**:

1. Обновить `CONTEXT.md` при изменении поведения.
2. **git commit** (сообщение — «зачем»).
3. **git push** в `origin/main` (деплой Cloud).

Исключения: черновик на полпути, «не коммить», секреты. Protected branch — ждать подтверждения в Cursor.
